# Subagent Usage Guide

## Agent Categories by Development Aspect

### Research
Agents for understanding, exploring, and initializing codebases:
- **external-context-researcher**: Deploy when integrating external APIs, libraries, or unfamiliar frameworks to gather comprehensive documentation and implementation guides. Creates research briefs that eliminate the need for additional browsing.
- **codebase-explorer**: Invoke for non-trivial tasks requiring understanding of existing code, patterns, or recent changes before proceeding. Generates targeted code snippets and architectural context to minimize token usage.
- **project-architect**: Use when starting a new project from scratch to scaffold proper directory structure and configuration files. Creates foundation before any feature work begins.

### Documentation
Agents for creating and maintaining project documentation:
- **docs-weaver**: Deploy after feature implementation or API changes to generate/update documentation with verified code examples extracted from tests. Validates link integrity and maintains documentation-code alignment.
- **project-historian**: Invoke after major structural changes (>500 lines, migrations, refactors) to create checkpoints with differential narratives. Documents what changed, architectural implications, and risk assessments with semantic tags.

### UI
Agents for user interface testing and copy optimization:
- **browser-navigator**: Use for automated end-to-end UI testing of web applications running locally. Tests interactive elements, validates layouts, and autonomously repairs obvious runtime bugs using Playwright MCP exclusively.
- **ux-copy-brainstormer**: Engage when creating/refining user-facing copy for interfaces, error messages, or onboarding flows. Requires brand voice guidelines; generates clarity-focused alternatives with rationale.

### Backend
Agents for backend optimization and data management:
- **migration-planner**: Engage when modifying database schemas, ORM models, or planning storage migrations. Designs zero-downtime strategies with comprehensive safety validations and rollback procedures.
- **cache-strategy-architect**: Engage when profiling reveals repeated expensive operations or when designing high-traffic features. Creates multi-layered caching architectures with TTL strategies, invalidation patterns, and observability metrics.
- **performance-profiler**: Deploy when latency budgets are exceeded or resource usage spikes to identify bottlenecks through profiling. Delivers minimal, high-ROI code patches with quantified performance improvements.

### Testing
Agents for test creation and validation:
- **backend-test-guardian**: Trigger after implementing backend features to create integration/unit tests or diagnose CI test failures. Maintains testing infrastructure and generates healing prompts for broken tests.
- **pre-push-validator**: Run before pushing code to verify style, linting, type safety, tests, and build integrity across all dimensions. Provides comprehensive pass/fail report with auto-fix capabilities for safe issues.

### Security
Agents for security scanning and credential management:
- **secrets-env-auditor**: Invoke before commits, pushes, or deployments to scan for exposed credentials and validate .env.sample completeness. Generates security audit reports with masked findings and rotation recommendations.
- **security-scanner**: Use after auth/authz changes, dependency updates, or for scheduled comprehensive scans (SAST, CVE scanning, config validation). Produces severity-classified findings with specific remediation steps.

### CI/CD
Agents for pipeline optimization:
- **cicd-optimizer**: Trigger when CI/CD execution time exceeds targets or shows degradation trends. Analyzes bottlenecks and creates optimization plans (parallelization, caching, affected-only builds) while preserving test coverage.

## Recursive Agent Chaining Patterns

### Pattern 1: Full Feature Lifecycle
```
codebase-explorer → [implement feature] → backend-test-guardian → pre-push-validator → security-scanner → docs-weaver → project-historian
```

### Pattern 2: External Integration
```
external-context-researcher → [implement integration] → backend-test-guardian → secrets-env-auditor → pre-push-validator
```

### Pattern 3: Database Evolution
```
codebase-explorer → migration-planner → [execute migration] → backend-test-guardian → project-historian
```

### Pattern 4: Performance Optimization
```
performance-profiler → cache-strategy-architect → [implement optimizations] → backend-test-guardian → pre-push-validator → project-historian
```

### Pattern 5: Security Hardening
```
security-scanner → [remediate issues] → secrets-env-auditor → pre-push-validator → docs-weaver
```

### Pattern 6: UI Development & Validation
```
ux-copy-brainstormer → [implement UI] → browser-navigator → docs-weaver → pre-push-validator
```

| **Documentation** | docs-weaver | Feature shipped, API changed, docs audit needed | Updated docs with verified code examples from tests |
| **Documentation** | project-historian | >500 LOC changed, architectural refactor, milestone | Checkpoint with differential narrative and risk assessment |
| **UI** | browser-navigator | UI testing needed, layout validation, debugging breakages | UI test report with screenshots, console logs, bug fixes |
| **UI** | ux-copy-brainstormer | Creating/refining user-facing copy, confusion reports | Copy alternatives with clarity improvements and brand alignment |
| **Backend** | migration-planner | Schema changes, ORM updates, storage migrations | Migration plan with safety validations and rollback scripts |
| **Backend** | cache-strategy-architect | Repeated expensive operations, high-traffic feature planning | Multi-layered cache architecture with TTL, invalidation, and metrics |

## Critical Decision Framework

**Use codebase-explorer when:**
- You need to understand existing patterns before implementing
- Task requires locating code across multiple files
- Recent git changes may impact your work

**Use external-context-researcher when:**
- Integrating something unfamiliar that requires web research
- Need official documentation for external dependencies
- Adopting new language/framework/MCP server

**Chain multiple agents when:**
- Task spans multiple phases (research → implement → test → document)
- Quality gates require validation from different domains
- Complex features need comprehensive coverage

**Skip agents when:**
- Task is trivial (<3 steps, single file)
- You already have complete context
- Changes don't warrant validation overhead

## Pro Tips for Recursive Usage

1. **Start broad, narrow down**: Begin with explorers/researchers, then move to implementers and validators
2. **Batch validation**: Run pre-push-validator + security-scanner + secrets-env-auditor together before pushing
3. **Document as you go**: Trigger docs-weaver immediately after features, not as an afterthought
4. **Checkpoint strategically**: Use project-historian at meaningful milestones, not after every small change
5. **Parallelize when possible**: Security, performance, and doc agents can often run concurrently
6. **Trust the specialists**: Each agent is optimized for its domain - let them work autonomously
7. **Chain intentionally**: Understand dependencies between agents (e.g., tests must pass before docs can extract examples)

---

## Common Anti-Patterns to Avoid

- Using browser-navigator for research tasks (it's for UI testing only)
- Skipping codebase-explorer on complex tasks (leads to inefficient context gathering)
- Running project-historian too frequently (creates noise)
- Ignoring pre-push-validator recommendations (technical debt accumulates)
- Using ux-copy-brainstormer without brand voice context (wastes effort)
- Deploying migrations without migration-planner review (data loss risk)

## Emergency Response Protocols

**Production incident?**
```
codebase-explorer → [diagnose] → [hotfix] → pre-push-validator → security-scanner → project-historian
```

**Security vulnerability discovered?**
```
security-scanner → secrets-env-auditor → [remediate] → backend-test-guardian → pre-push-validator
```

**CI pipeline broken?**
```
backend-test-guardian [diagnose failures] → [fix] → pre-push-validator → cicd-optimizer [if chronic]
```

**Performance degradation?**
```
performance-profiler → [optimize] → backend-test-guardian [verify] → project-historian [document]
```
