# Frontend

This is the merged frontend skill for the repo.

It combines:

- design direction and originality
- practical React and Next.js implementation
- layout and typography systems
- performance and bundle discipline
- frontend review and refinement

## Modes

### 1. Design direction

Use when the problem is mostly aesthetic:

- page look and feel
- layout direction
- typography choices
- motion and visual hierarchy
- avoiding generic AI-looking UI

### 2. Implementation

Use when the problem is mostly engineering:

- component scaffolding
- stateful UI
- responsive behavior
- Next.js or React patterns
- performance and bundle cleanup

### 3. Review and polish

Use when the UI exists and needs tightening:

- visual consistency
- spacing and composition
- accessibility and interaction states
- responsiveness
- code and design drift

## Core workflow

Use this order:

`PURPOSE -> AESTHETIC -> SYSTEM -> BUILD -> OPTIMIZE -> QA`

### Purpose

Understand:

1. who uses the interface
2. what job it must do
3. what action matters most
4. what technical constraints exist

### Aesthetic

Pick a real point of view.

Examples:

- editorial
- brutalist
- refined minimal
- industrial
- playful
- luxury
- retro-futurist
- soft and organic

Do not default to safe, generic UI patterns.

### System

Define:

- type scale
- spacing scale
- color system
- component rules
- interaction patterns
- mobile behavior

### Build

Prefer clean, production-grade implementation:

- semantic structure
- predictable component boundaries
- responsive layouts first
- accessible states
- minimal unnecessary abstraction

### Optimize

Check:

- bundle weight
- image strategy
- font loading
- motion cost
- layout shift
- rendering hotspots

### QA

Before shipping, verify:

- hierarchy is obvious
- CTA is clear
- mobile layouts work
- keyboard and focus states exist
- edge-case content does not break layout
- the interface still feels intentional after implementation

## Design rules

- clarity first
- consistency builds trust
- mobile constraints come first
- motion should guide, not decorate
- content should drive layout decisions

## Engineering rules

- keep components understandable
- optimize where measurement says it matters
- follow framework conventions instead of fighting them
- treat accessibility as a default requirement
- use comments sparingly and only where the logic is not obvious
